CONFIRMED PROBLEM
==================================================

Repository evidence indicates that most intentionally public restaurant routes use a restricted public projection, but one route does not.

The affected branch is reported as:

GET /api/restaurants?action=neonRestaurant

or the corresponding Vercel rewrite for:

/api/neon/restaurant/:id

This branch can return the raw database restaurant row without requiring restaurant membership or superadmin authorization.

Depending on the current schema, the raw row may expose internal fields such as:

- owner_id;
- internal status;
- subscription plan;
- plan limits;
- internal lifecycle timestamps;
- deletion state;
- operational flags;
- compatibility fields;
- internal storage keys;
- fields added to the table in the future.

Returning the complete database row also creates a future-data-leak risk because newly added columns become public automatically.

==================================================

5. Sensitive or operational fields cannot appear in public responses, including when new database columns are added later.

6. Cross-tenant access to internal restaurant records is denied.

7. Existing legitimate public restaurant and menu functionality remains operational.

STEP 2 — DEFINE EXPLICIT RESTAURANT DTO CONTRACTS
==================================================

Create or reuse explicit server-side DTO builders.

At minimum, define:

1. PublicRestaurantDTO

Used by unauthenticated restaurant and public-menu routes.

2. MemberRestaurantDTO

Used by authenticated restaurant members where additional operational fields are legitimately required.

3. SuperadminRestaurantDTO

Used only for verified platform-superadmin operations when the complete administrative view is required.

Do not return the raw database object even to superadmin routes unless there is a documented reason and a validated schema.

--------------------------------------------------
PUBLIC DTO REQUIREMENTS
--------------------------------------------------

The public DTO must use an explicit allowlist.

Include only fields genuinely required by the public menu or public restaurant website, such as where supported by the existing product:

- public ID or public UID;
- slug;
- restaurant name;
- logo or safe public image URL;
- cover image;
- public description;
- address or public location;
- public contact information intentionally displayed;
- opening hours;
- cuisine/category;
- public social links;
- public booking or ordering availability;
- public theme or presentation settings;
- safe table-display context where intentionally public.

Do not add a field merely because it exists in the database.

--------------------------------------------------
PUBLIC DTO FORBIDDEN FIELDS
--------------------------------------------------

The public response must never include fields such as:

- owner_id;
- owner email;
- membership data;
- internal user IDs;
- internal plan or subscription state;
- plan_limits;
- billing identifiers;
- payment identifiers;
- internal notes;
- deletion flags;
- deleted_at;
- lifecycle audit fields not required publicly;
- private configuration;
- secret or credential fields;
- internal storage object keys;
- database connection information;
- private phone or email fields not explicitly intended for display;
- internal permissions;
- staff information;
- raw JSON configuration containing private values.

The test must detect forbidden fields recursively, not only at the top level.

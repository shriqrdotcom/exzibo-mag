==================================================
CONFIRMED PROBLEMS
==================================================

Repository evidence indicates:

1. Preview-token signing can fall back through:

   PREVIEW_SECRET
   → REPL_ID
   → literal "preview-hmac-secret"

2. A missing secret can therefore result in a predictable signing key.

3. Signature comparison may use normal string equality rather than a timing-safe comparison.

4. Preview endpoints may be registered inside a general deployable Express runtime.

5. Preview authentication may not have a strict issuer, audience, lifetime and runtime boundary.

6. It is not yet proven whether the preview-authentication system is still required.

==================================================
OBJECTIVE
==================================================

First determine whether the custom preview-authentication flow is actively required.

Then choose exactly one path:

A. Remove the preview-authentication flow when it is unused.

or:

B. Retain it only in a dedicated preview runtime and secure its complete token lifecycle.

After this prompt:

1. No preview signing key falls back to REPL_ID or a literal.

2. Preview endpoints are unavailable in normal production and general deployment runtimes.

3. Preview authentication cannot automatically grant superadmin authority.

4. Token validation is strict and timing-safe when the flow is retained.

5. Preview secrets are explicitly configured and fail closed.

STEP 2 — SELECT ONE IMPLEMENTATION PATH
==================================================

PATH A — REMOVE UNUSED PREVIEW AUTHENTICATION

Use this path only when repository evidence proves the flow is unused.

Required work:

1. Remove preview-login and preview-verify routes.

2. Remove preview-token generation and verification helpers that have no other caller.

3. Remove dead frontend callers.

4. Remove preview-authentication cookies or localStorage behaviour.

5. Remove obsolete preview-auth environment documentation.

6. Remove obsolete tests and replace them with a regression test proving the retired routes are unavailable.

7. Do not remove unrelated Replit deployment or preview functionality.

PATH B — RETAIN AND SECURE PREVIEW AUTHENTICATION

Use this path when the flow is actively required.

Required work:

1. Register preview routes only when a server-only runtime marker explicitly enables them, for example:

   APP_RUNTIME=preview

2. Do not enable preview authentication using a browser-exposed VITE variable.

3. Preview routes must not register in:

   - production;
   - normal local development;
   - general Replit deployment;
   - tests unless explicitly enabled;
   - Vercel production.

4. PREVIEW_SECRET must be mandatory whenever preview routes are enabled.

5. Remove every fallback to:

   - REPL_ID;
   - preview-hmac-secret;
   - another literal;
   - a random runtime value.

6. Validate a reasonable minimum PREVIEW_SECRET length during startup.

7. Fail startup or preview-route initialization when required configuration is missing.

8. Never log the preview secret, token or signature.

==================================================
STEP 3 — SECURE TOKEN FORMAT WHEN RETAINED
==================================================

When PATH B is selected, implement a strict token contract containing at minimum:

- version;
- subject;
- issued-at;
- expiry;
- issuer;
- audience;
- token ID.

Requirements:

1. Use a clear versioned token format.

2. Use a cryptographically secure token ID:

   - randomBytes;
   - randomUUID;
   - UUID from a cryptographically secure implementation.

3. Use a short documented maximum lifetime.

4. Reject tokens with excessive lifetime.

5. Reject issued-at values too far in the future.

6. Allow only a small documented clock-skew tolerance.

7. Require exact issuer and audience values.

8. Reject unknown token versions.

9. Reject missing claims.

10. Reject malformed base64.

11. Reject malformed JSON.

12. Reject unexpected claim types.

13. Do not trust claims until the signature is verified.

==================================================
STEP 4 — TIMING-SAFE SIGNATURE VERIFICATION
==================================================

When PATH B is selected:

1. Calculate the expected signature using the explicit PREVIEW_SECRET.

2. Decode supplied and expected signatures into byte buffers.

3. Confirm both buffers have equal length before comparing.

4. Use:

   crypto.timingSafeEqual

5. Reject invalid encoding cleanly.

6. Do not return cryptographic details in client responses.

7. Do not reveal whether the subject, email, password hash or signature was the specific failed element.

8. Return one stable public authentication failure.

==================================================
STEP 5 — PREVIEW LOGIN PROTECTION
==================================================

When PATH B is selected:

1. Keep or add rate limiting.

2. Apply a strict request-body size limit.

3. Reject unknown body fields.

4. Normalize and validate email or subject input.

5. Use constant-time comparison where sensitive fixed-length values are compared.

6. Never store or return plaintext passwords.

7. Never grant superadmin merely because preview authentication succeeded.

8. Any preview role must be:

   - explicit;
   - minimally privileged;
   - documented;
   - server-controlled.

9. Prefer a restricted read-only preview identity where product requirements allow it.

==================================================
STEP 6 — COOKIE OR TOKEN STORAGE
==================================================

2. Set Secure outside localhost.

5. Use a bounded Max-Age matching the token lifetime.

6. Do not expose the token to frontend JavaScript unless the flow strictly requires it.

7. Do not store preview tokens in localStorage.

8. Clear the cookie on invalid or expired verification where appropriate.

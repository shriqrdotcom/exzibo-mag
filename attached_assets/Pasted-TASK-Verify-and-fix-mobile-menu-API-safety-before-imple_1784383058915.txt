GOAL

Make the existing menu APIs safe for use by the Expo mobile application without changing the public customer menu.

VERIFY AND FIX

1. Item partial updates

Ensure the mobile item update endpoint supports a real partial update.

- Omitted fields must remain unchanged.
- Do not replace omitted values with defaults.
- Resolve the item’s restaurant ID from the database.
- Do not trust restaurantId supplied by the client.
- Availability and published toggles must not require resending the complete item.

2. Category tenant isolation

For an existing category ID:

- resolve its restaurant from the database
- verify authenticated membership and Manager-or-higher permission
- reject a submitted restaurantId that does not match
- prevent Restaurant A from modifying Restaurant B’s category

3. Item and bulk-upsert isolation

For every existing item ID:

- resolve restaurant ownership from the database
- verify access before updating
- validate every item independently in bulk requests
- reject the complete request safely if any item is unauthorized

4. Category relationship

When an item contains categoryId:

- verify the category exists
- verify it belongs to the same restaurant as the item
- reject cross-restaurant category assignment

5. Role separation

Menu mutations may be performed only by:

- owner
- admin
- manager

Staff must receive 403.

Do not allow menu_studio or superadmin to bypass normal restaurant authorization. Superadmin access must remain separate.

6. Upload validation

Verify that menu uploads:

- require an authenticated Manager-or-higher restaurant member
- validate actual decoded image bytes
- reject unsupported or malformed files
- enforce file-size and reasonable dimension limits
- use server-generated R2 keys
- never trust client filenames or MIME labels

Report separately whether image replacement leaves orphaned R2 objects. Do not build a large media redesign in this task.

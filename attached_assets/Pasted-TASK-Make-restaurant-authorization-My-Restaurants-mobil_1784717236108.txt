PROBLEM

The repository currently uses different identity rules:

- protected restaurant APIs use email;
- My Restaurants uses user_id;
- mobile bootstrap uses user_id with email fallback.

This can make the same user receive different restaurant access in different parts of the application.

GOAL

Use this rule everywhere:

1. Match the authenticated Better Auth user_id first.
2. Use normalized email only when the membership row has user_id IS NULL.
3. Never allow an email match to override a different existing user_id.

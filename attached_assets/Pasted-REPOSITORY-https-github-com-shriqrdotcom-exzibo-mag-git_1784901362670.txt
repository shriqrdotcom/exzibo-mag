CONFIRMED PROBLEMS
==================================================

Repository evidence indicates:

1. Preview-token signing can fall back through:

   PREVIEW_SECRET
   → REPL_ID
   → literal "preview-hmac-secret"

2. Missing preview configuration can therefore result in a predictable signing key.

3. Preview-token verification may use ordinary string equality rather than a timing-safe comparison.

4. Preview routes may be registered inside the same Express application used by a deployable runtime.

5. Preview tokens may lack a strict:

   - version;
   - issuer;
   - audience;
   - maximum lifetime;
   - cryptographically secure token ID;
   - dedicated runtime boundary.

6. Preview authentication may grant more authority than is necessary.

7. It has not yet been conclusively established whether the preview-authentication flow is still used.

PATH B — RETAIN AND SECURE PREVIEW AUTHENTICATION
==================================================

Use PATH B when the flow is still required or cannot safely be removed.

--------------------------------------------------
B1 — DEDICATED RUNTIME BOUNDARY
--------------------------------------------------

1. Register preview routes only when a server-only runtime setting explicitly enables them.

Use a validated server-only variable such as:

   APP_RUNTIME=preview

2. Do not enable preview authentication using:

   - VITE_PREVIEW_MODE;
   - another browser-exposed VITE variable;
   - REPL_ID;
   - hostname inference;
   - automatic Replit detection.

3. Preview routes must not register in:

   - production;
   - normal development;
   - general Replit deployment;
   - Vercel production;
   - tests unless explicitly enabled by the test setup.

4. The default behaviour must be:

   preview routes disabled

5. When APP_RUNTIME=preview is selected, required preview configuration must be validated before the server starts accepting requests.

--------------------------------------------------
B2 — EXPLICIT SECRET POLICY
--------------------------------------------------

1. PREVIEW_SECRET must be explicitly supplied whenever preview routes are enabled.

2. Remove all fallback behaviour involving:

   - REPL_ID;
   - preview-hmac-secret;
   - another literal;
   - generated runtime secrets;
   - hostnames;
   - deployment IDs.

3. Validate the preview secret at startup.

4. Require a minimum length of at least 32 characters.

5. Reject empty, placeholder or obviously unsafe values.

6. Never log the secret.

7. Never include the secret in:

   - errors;
   - HTTP responses;
   - test reports;
   - documentation;
   - committed files.

8. Missing or invalid PREVIEW_SECRET must prevent preview-auth route activation.

--------------------------------------------------
B3 — VERSIONED TOKEN CONTRACT
--------------------------------------------------

Create one strict token contract containing:

- version;
- subject;
- issuedAt;
- expiresAt;
- issuer;
- audience;
- tokenId.

Recommended logical structure:

{
  "version": 1,
  "subject": "...",
  "issuedAt": 1234567890,
  "expiresAt": 1234568190,
  "issuer": "exzibo-preview",
  "audience": "exzibo-preview-access",
  "tokenId": "cryptographically-secure-id"
}

Requirements:

1. Use a documented token version.

2. Reject unknown versions.

3. Generate tokenId using a cryptographically secure source.

4. Use integer Unix timestamps consistently.

5. Use a short maximum lifetime.

Recommended maximum:

   5 to 15 minutes

6. Reject tokens whose lifetime exceeds the configured maximum.

7. Reject expired tokens.

8. Reject issuedAt values beyond a small allowed clock-skew window.

9. Require exact issuer and audience values.

10. Reject missing claims.

11. Reject incorrect claim types.

12. Reject unexpected token structure when strict schema validation is available.

13. Do not trust token claims until the signature has been verified.

--------------------------------------------------
B4 — SIGNING AND TIMING-SAFE VERIFICATION
--------------------------------------------------

1. Use HMAC-SHA256 or the repository’s existing secure equivalent.

2. Sign one canonical encoded payload representation.

3. Use base64url consistently.

4. During verification:

   - parse the token structure safely;
   - validate the encoded sections;
   - calculate the expected signature;
   - decode the supplied signature into a Buffer;
   - decode the expected signature into a Buffer;
   - confirm both buffers have equal length;
   - compare using crypto.timingSafeEqual.

5. Never use ordinary string equality for signature comparison.

6. Reject invalid encoding without throwing internal details to the client.

7. Return one stable public error for invalid credentials.

8. Do not reveal whether failure was caused by:

   - email;
   - password;
   - token format;
   - expiry;
   - signature;
   - issuer;
   - audience.

9. Log only a safe failure category and request ID.

--------------------------------------------------
B5 — PREVIEW LOGIN PROTECTION
--------------------------------------------------

1. Keep or add rate limiting to preview login.

2. Apply a strict request-body size limit.

3. Use an explicit HTTP method.

4. Reject unsupported methods with 405.

5. Validate the request body using a strict schema.

6. Reject unknown fields.

7. Normalize email input where applicable.

8. Do not store or return plaintext passwords.

9. Do not log supplied credentials.

10. Use a generic authentication-failure response.

11. Do not grant automatic superadmin access after preview login.

12. Prefer a restricted preview identity with only the minimum permissions necessary.

13. Any preview role must be:

    - server-controlled;
    - explicit;
    - documented;
    - limited;
    - auditable.

14. Preview authentication must not bypass restaurant tenant checks.

--------------------------------------------------
B6 — COOKIE OR TOKEN STORAGE
--------------------------------------------------

When the preview token is stored in a cookie:

1. Set HttpOnly.

2. Set Secure outside localhost.

3. Set an explicit Path.

4. Set a bounded Max-Age matching the token lifetime.

5. Use the narrowest appropriate SameSite policy.

6. Do not expose the token to frontend JavaScript.

7. Do not store the preview token in localStorage.

8. Do not store it in sessionStorage unless the architecture strictly requires it and the security implications are documented.

9. Clear invalid or expired preview cookies where appropriate.

10. Do not mix the preview token with the Better Auth session cookie.
